# YACSエージェント

Claude Codeの専門エージェント。各エージェントは自分の分野について深い知識を持ち、ベストプラクティスを実装します。

## エージェントとは？

エージェントは以下の特徴を持つ永続的な専門家です：
- 最初の呼び出しでプロジェクト構造を学習
- セッション内の会話全体で専門知識を保持
- 明示的に呼び出すか、Claude Codeによって自動的に活性化できる
- 実際のコードに基づいたレコメンデーションを提供
- ドメイン固有の原則とパターンを実装

スキル（ワンショットコマンド）とは異なり、エージェントはコンテキスト認識で状態を持っています。

## エージェント

### 🏢 バックエンドエキスパート (`backend-expert/`)
API、サービス、データモデル、認証、パフォーマンスなどサーバーサイドアーキテクチャを専門としています。

**使用場面：**
- APIデザインまたはエンドポイント契約のレビュー
- データベーススキーマまたはデータアクセスパターンの設計
- 認証、認可、またはセキュリティミドルウェアの実装
- クエリ、キャッシング戦略、または非同期パターンの最適化
- サービスまたはリポジトリのバグ調査

**主な原則：**
- コントローラーにビジネスロジックなし；すべてのロジックはサービスにある
- バリデーション境界で実行（サービス前の入力検証）
- ORMが利用可能な場合は生クエリなし
- シークレットはハードコードなし；常に環境変数で管理

### 🎨 フロントエンドエキスパート (`frontend-expert/`)
UI、コンポーネント、状態管理、ルーティング、スタイリング、アクセシビリティ、パフォーマンスを専門としています。

**使用場面：**
- コンポーネント階層またはPropsAPIの設計
- バリデーションとエラー状態を持つフォームの実装
- 状態管理の設定（Context、Redux、Zustandなど）
- パフォーマンスの最適化（遅延ロード、メモ化、コード分割）
- アクセシビリティ（a11y）標準の確保
- スタイリングアプローチと応答型デザインのレビュー

**主な原則：**
- コンポーネントはUIをレンダリングし、相互作用を処理；ビジネスロジックはhooks/storesに
- 2レベルを超えるプロップドリリングなし；代わりにcontextまたはstoresを使用
- すべてのデータフェッチはloading、error、successの状態が必須
- アクセシビリティは非交渉；キーボードナビゲーションとスクリーンリーダー対応が必須
- コードベース全体の一貫したパターン

### 🔐 セキュリティエキスパート (`security-expert/`)
すべてのレイヤーにおけるセキュリティ脆弱性、設定ミス、セキュリティアンチパターンを監査します。

**使用場面：**
- マージ前のセキュリティレビュー
- リリースまたはデプロイメント前
- 認証、認可、または暗号化を実装する場合
- サードパーティ依存関係またはサプライチェーンのレビュー
- 新機能前の脅威モデリング
- インシデント調査

**主な原則：**
- シークレットはコード、ログ、エラーレスポンスに絶対にない
- 認可チェックはデフォルトで拒否（deny > allow）
- すべての入力をシステム境界で検証
- 検証済みの暗号化のみ使用（自作なし）
- エラーは安全に失敗し、内部を公開しない
- 多層防御：単一障害点なし

### ☁️ インフラストラクチャエキスパート (`infra-expert/`)
デプロイメント、CI/CD、コンテナ化、IaC、クラウドリソース、運用を専門としています。

**使用場面：**
- CI/CDパイプラインの設計またはレビュー
- DockerfileとComposeセットアップの構築
- Terraform、Pulumi、またはその他のIaCの作成
- Kubernetesマニフェストの設定
- 可観測性の設定（ロギング、メトリクス、トレース）
- 環境設定とシークレット管理
- 信頼性、スケーリング、災害復旧のための設計

**主な原則：**
- シークレットはシークレットマネージャーで管理；イメージまたはコードに絶対にない
- イミュータブルインフラストラクチャ：変更ではなく置き換え
- 環境はコードから再現可能である必要がある
- ヘルスチェック（liveness、readiness）は必須
- IAM、サービスアカウント、ネットワークポリシーで最小権限原則
- パイプラインは高速に失敗：コスト高いステップ前にlintと単体テスト

### 🧪 QAエキスパート (`qa-expert/`)
テスト戦略、カバレッジ分析、テスト品質、バグ調査を専門としています。

**使用場面：**
- テスト戦略の設計（各レベルで何をテストするか）
- カバレッジギャップの分析
- テスト失敗またはフレーキネスの調査
- テストコードの品質とメンテナンス性のレビュー
- 新機能のテスト設計
- 報告されたバグの根本原因分析

**主な原則：**
- テストは実装詳細ではなく動作を検証
- 各テストは1つの失敗理由を持つ
- テストピラミッド：unit >> integration >> E2E
- 外部システムに対するMock、自分のコードに対してはなし
- 修正されたすべてのバグが回帰テストを得る
- カバレッジはゴールではなく最小値（100%の悪いテスト < 80%の良いテスト）

## インストール

### オプション1：プロジェクトにコピー

個別のエージェントまたはすべてを `.claude/agents/` にコピー：

```bash
# 1つのエージェントをコピー
cp -r YACS/agents/backend-expert ~/.claude/agents/

# すべてのエージェントをコピー
cp -r YACS/agents/* ~/.claude/agents/
```

### オプション2：YACSリポジトリから参照

YACSをローカルに複製している場合、コピーなしで直接Claude Codeのエージェント パスを参照できます。

## エージェントの使用

### 明示的な呼び出し

Claude Codeに特定のエージェントを使用するように依頼：

```
@backend-expert このユーザーサービスをN+1クエリでレビュー
infra-expertを使ってこのDockerfileを分析してください
security-expert、この認証フローを監査してください
qa-expert、この機能に何のテストが不足していますか？
```

### プロアクティブな活性化

Claude Codeはコンテキストに基づいて関連エージェントを自動的に活性化できます。例えば：
- APIデザインについての質問は `backend-expert` をトリガー
- コンポーネント アーキテクチャの議論は `frontend-expert` をトリガー
- Kubernetesファイルの確認は `infra-expert` をトリガー
- セキュリティの懸念は `security-expert` をトリガー
- テストについての議論は `qa-expert` をトリガー

## エージェント構造

各エージェントは以下を含む単一の `AGENT.md` ファイルです：

```yaml
---
name: <agent-name>
description: <このエージェントをトリガーするもの>
---

## Onboarding
[エージェントがプロジェクトを学習する方法]

## Expertise
[エージェントがカバーしているもの]

## How to respond
[応答ガイドライン]

## Principles
[実装するドメイン固有の原則]
```

`description` フィールドはClaude Codeにエージェント活性化を検討する時期を示します。

## カスタムエージェントの作成

`.claude/agents/` ディレクトリでプロジェクト固有のエージェントを作成できます。同じ `AGENT.md` 形式に従う：

1. `~/.claude/agents/<agent-name>/AGENT.md` を作成
2. `name`、`description` を含むYAMLフロントマターを使用
3. onboarding、expertise、応答ガイドライン、原則を含める
4. Claude Codeが自動的にエージェントを検出して提供します

例：プロジェクト固有のデータレイヤー用の `database-expert`、またはチームのスタイルガイドを実装する `team-conventions-agent`。

## 注記

- すべてのエージェント定義はYAMLフロントマター付きの `.md` ファイル
- エージェントはプロジェクト認識：アドバイス前にコードベースを読む
- 各エージェントは `description` で定義された特定のトリガーを持つ
- エージェントはClaude Codeの組み込み推論と連携
- YACSのエージェントとプロジェクト固有のカスタムエージェントを必要に応じてミックス
